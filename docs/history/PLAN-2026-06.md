# ビルド計画

## 数学的アプローチ（定石）

1D-CSPの王道は **Gilmore-Gomory の列生成（column generation）**:

- **マスター問題（LP緩和）**: 各カットパターンの使用本数の和を最小化、各需要長の充足を制約とする。
- **pricing subproblem**: マスターの双対価格を価値とする**ナップサック問題**を解き、被約費用が負の新パターンを生成して列に追加。負の列が無くなれば停止。
- **整数解**: 1Dは整数ギャップが小さいのでLP解の切り上げで実用上ほぼ最適。厳密に詰めるなら **branch-and-price**。

実装ライブラリ: OR-Tools（LP+ナップサックループを自前で回す、または小〜中規模ならCP-SATで直接定式化）。無料で殴れる代替: HIGHS、SCIP（branch-and-price向き）。

## フェーズとウルトラコードのタイミング

撃つべきは**広くて間違えると高くつく分岐だけ**（太字3箇所）。乱射するな。

| フェーズ | ソロ or ウルトラ | 内容 |
|---|---|---|
| 雛形・git・依存セットアップ | ソロ | 機械作業 |
| **ソルバ核の設計** | **ウルトラコード** | アーキ・ライブラリ選定・多目的(パレート)の組み方を独立複数案で競わせ、判断パネルで統合 |
| ソルバ核の実装 | ソロ | 設計が決まれば一本道 |
| **ソルバの正当性検証** | **ウルトラコード** | 既知ベンチマーク＋退化入力（kerf境界・需要0・1本超過長など）で敵対的に最適性を裏取り |
| GUI設計（パレート前線の見せ方） | ウルトラ寄り（任意） | 「充実GUI」狙いなら独立デザイン案を競わせる |
| GUI実装・調整 | ソロ | playwright/chrome-devtoolsで起動→スクショ→修正の輪を回す |
| **最終レビュー** | **ウルトラコード** | 正確性・性能・エッジ・UIを次元分けして発見→敵対検証 |

運用: **ウルトラコードはセッション全体でONにしない**。該当メッセージにだけ「ウルトラコード」と添える（例:「ソルバの設計を詰めろ、ウルトラコードで」）。それ以外はソロが速い。

冷や水: この問題の数学は教科書通り（Gilmore-Gomoryは定石）。文献漁りの価値は薄い。ウルトラコードの見返りは **(a) アーキ/GUIの設計を複数案で競わせる、(b) ソルバが本当に最適解を吐くかを敵対的に裏取りする** の二点に集中。
