## Wikipedia古典例での検証（2026-06-18）

投入: master=5600, kerf=0, 13品目219本（既知最適 73本 / 廃棄1640 / 0.401% / 最少種類10）

- **材料軸（arc-flow+HiGHS）= 満点**。73本/1640/0.401%/proven_optimal=true で既知最適と完全一致。健全、確定。
- **段取り軸（CP-SAT 設定モデルB）= 欠陥あり**（下記）。

### 段取り軸の真の欠陥（実測で確定）

過去に「13を setup_proven=True で誤証明する」と報告していたが、**それはハルシネーション**。実コードは `proven = OPTIMAL and bound==objective`（setup_mip.py）で、嘘の証明はしない。実測の本当の症状はこうだった:

1. **非再現・空出力**: `R=bars`（73スロット）×非凸整数積でCP-SATが重すぎ、90〜300秒で実行可能解すら見つけられず status=UNKNOWN を返す。`pareto.py` がその点を黙ってスキップ → 実行のたびに「13が出る／空」が揺れる。
2. **退行**: 設計（SOLVER_DESIGN:74）は「R = types を初期値」と規定。出荷コードの `R = bars` はそこからのドリフト。

## A: ロバスト化（実装済み・このブランチ feat/setup-axis-robustness）

`setup_mip.py` / `pareto.py` に3点:

- **warm-start**: 材料解パターンを初期可行点としてCP-SATにヒント注入（UNKNOWN/空を封じる）
- **R縮約**: スロット上限を材料解の種類数 P_mat に戻す（最適 P ≤ P_mat なので証明は保たれる＝安全な縮約。設計意図に復帰）
- **フォールバック anchor**: ソルバが解を返せない／種より悪い時は材料解を採用、点を絶対に落とさない

### 結果（実測）

| 観点 | 修正前 | 修正後 |
|------|--------|--------|
| 再現性 | コイントス | **完全再現**（run1=run2） |
| 空出力 | UNKNOWNで点消失 | **必ず点が出る** |
| z=73 の P | 13（運次第） | **12** |
| 前線 | 壊れ | z=73→12 / z=74→8 の本物のトレードオフ |
| テスト | 55 green | **55 green 維持** |

## B: 再定式化（実装済み）= 候補プール選択MIP

config-B が中規模でも最適に届かない（z=73で12止まり/proven立たず）ため、設計(SOLVER_DESIGN:85-87)記載の**候補プール選択MIP**を主経路に実装（`setup_mip.py`）:

- maximal パターンを全列挙（`_enumerate_maximal_patterns`）
- 使用本数 x_q・使用フラグ y_q の線形MIP（`Σx==bars`, `Σ a·x ≥ d`, `min Σy`, `num_workers=1`で決定論）
- maximal 限定でも最適性不変（非maximalは含むmaximalへ同一本数で置換可）＝全列挙が尽きれば真の最小に一致し proven 正当。列挙 cap(5万) 超は config-B へ退避

### 結果（実測）

| z | types | setup_proven | status |
|---|-------|--------------|--------|
| 73 | **10** | **True** | OPTIMAL |
| 74 | 8 | True | OPTIMAL |

**z=73 が既知最適10に proven 到達**（config-Bの12止まりを突破）。バイト完全再現。回帰テスト `solver/tests/test_benchmark.py`（Wikipedia既知最適突き合わせ, slow marker）追加。既定スイート57 green。
